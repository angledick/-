## 一、架构演进总览

### 阶段 1：历史原型（仅思路探讨，未采用）

```
用户 ──→ Chat UI (React) ──→ FastAPI
                                    │
                              ┌─────┴─────┐
                              │  Codex SDK │ ← skills/ + MCP tools(6)
                              └─────┬─────┘
                                    │
                          ┌─────────┼─────────┐
                          ▼         ▼         ▼

                          │         │         │
```

**历史原型特征：**
- 前端仅模拟业务流程，所有数据为 Mock
- 后端虽有 API 路由，但合规逻辑为硬编码返回
- RAG 知识库从未真正初始化
- 第三方平台（Shopify/飞书）无真实集成
- 事件系统、定时任务、风险推送均为静态展示

## 五、关键设计决策

### 5.1 为什么用文件驱动（而非数据库）？

Agent/Worker/Event/Skill 的定义使用 Markdown/YAML 文件：
- ✅ 非工程师也能编辑（产品经理可写 Agent 定义）
- ✅ Git 版本控制天然支持
- ✅ 热加载：修改文件立即生效，无需重启
- ✅ 可读性远超数据库记录

### 5.2 为什么用四层存储？

| 存储 | 适用场景 | 选择理由 |
|------|---------|---------|
| SQLite | 结构化关系数据（订单/报关单） | 零部署成本，事务保证 |
| JSON | 配置/索引/缓存 | 读写简单，全量加载内存 |
| ChromaDB | 向量检索（RAG） | 语义相似度搜索 |
| Markdown | 知识/Agent 定义 | 人类可读，Git 友好 |

### 5.3 为什么用 EventBus 而非消息队列？

- ✅ 单进程内通信，无需额外中间件
- ✅ Python asyncio 原生，零延迟
- ✅ `on_all` 全局监听，Manager Agent 一处接管
- ✅ 对中小卖家场景，性能完全足够
