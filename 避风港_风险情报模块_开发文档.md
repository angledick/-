> 定位：在现有 `news_monitor` + `risk_alert` + `proactive_engine` 三套分离基础设施上，构建统一的**全球风险情报引擎**，覆盖关税/战争/金融三大风险因子域，支持用户主动关键词检索与周期性自动检索。

## 一、调研结论（先决条件）

### 1.1 现有可复用基础设施

| 模块 | 文件路径 | 可复用程度 | 说明 |
|------|---------|-----------|------|
| 新闻采集器 | `data/skills/news-collect/script/news_collector.py` | ⭐⭐⭐ 直接扩展 | 已有 Fed/WTO/BIS/ECB/金十/中国新闻网 6 个数据源；关键词评分框架完整 |
| 新闻分析器 | `data/skills/news-analyze/script/news_analyzer.py` | ⭐⭐⭐ 直接扩展 | LLM 批量分析框架；输出 `risk_direction/risk_level/affected_markets` |
| 新闻存储层 | `app/storage/news_store.py` | ⭐⭐⭐ 直接复用 | SQLite 3 表结构；已有 `news_keyword_config` 用户关键词表 |
| 米塔搜索 | `data/tools/impl/metaso_search.py` | ⭐⭐⭐ 直接复用 | 联网实时检索；Bearer Token 鉴权；标准 JSON 出参 |
| 风险预警存储 | `app/core/risk_alert.py` | ⭐⭐ 集成对接 | JSON 文件持久化；`create_alert/get_alerts/dismiss_alert` 接口完整；未与 EventBus 联通 |
| 事件总线 | `app/core/event_bus.py` | ⭐⭐⭐ 直接发布 | `publish_raw` 标准化入口；条件订阅 AST 引擎；已定义 8 类 risk 事件 |
| 定时调度器 | `app/core/scheduler.py` | ⭐⭐ 绑定新任务 | APScheduler + bindings.yaml；已有 `poll_all_markets/news_collect_and_analyze` 任务槽 |
| 新闻监控 API | `app/api/news_monitor.py` | ⭐⭐ 扩展端点 | 已有 `GET /news-monitor/news`、`POST /news-monitor/collect`、关键词 CRUD |
| 风险预警 API | `app/api/risk.py` | ⭐⭐ 扩展端点 | 已有预警列表/扫描/仪表盘；缺少全文检索和跨来源聚合 |
| 主动引擎 | `app/core/proactive_engine.py` | ⭐ 参考扩展 | 已有洞察/简报框架；风险评分字段为空 |

### 1.2 关键缺口（必须新建）

| 缺口 | 影响 | 优先级 |
|------|------|--------|
| **全球新闻源不足** | 现有 6 个源均为金融/贸易政策，缺地缘政治、战争、制裁、关税变动专项源 | P0 |
| **三大风险因子域分类** | 无 `tariff / conflict / financial` 三级体系；现有仅 `risk_direction(利多/利空/中性)` | P0 |
| **用户主动关键词检索** | 现有关键词仅用于新闻过滤评分，无法驱动实时联网搜索并返回结果 | P0 |
| **周期性检索任务管理** | `news_collect_and_analyze` 任务存在但无法按用户关键词动态生成子任务 | P1 |
| **跨来源风险聚合 API** | 预警(JSON文件) + 新闻(SQLite) + 事件(内存) 三处数据孤立，无统一检索入口 | P1 |
| **自动关键词推荐** | 无法根据用户产品的市场/HS编码自动推导检索关键词 | P2 |
| **风险情报与预警联动** | 新闻分析结果未自动触发 `risk_alert:new` 事件和预警写入 | P1 |

### 1.3 调研断点确认

- `risk_worker` 已在 `bindings.yaml` 中定义但 `execute_worker_task()` 无具体实现 → 新模块通过 `_execute_via_worker` 分发
- `news_keyword_config` 表存在但当前仅支持全局配置，未按用户分域 → 需扩展为用户级 + 关键词级配置
- `metaso_search.py` 仅支持 5-10 条结果、无分页 → 新增批量检索封装，循环查询不同关键词
- `EventBus` 的 `risk:threshold_breached` 等 8 个风险事件已定义但无生产方 → 本模块为主要事件发布者

## 二、模块定义

### 2.1 模块名称与边界

**模块名称：Risk Intelligence Engine（风险情报引擎）**  
**代码标识：** `risk_intel`  
**隶属层级：** 核心服务层（与 `proactive_engine`、`market_monitor` 同级）

**边界定义（IN scope）：**
- 全球主流媒体/官方数据源新闻采集（扩展现有 news-collect Skill）
- 三大风险因子域分类：`tariff`（关税）/ `conflict`（战争/制裁/地缘）/ `financial`（金融市场）
- 用户主动输入关键词 → 实时联网检索 → 结构化结果返回
- 周期性自动检索任务（按关键词/按产品域/按市场域）
- 自动关键词推荐（基于产品 HS 编码 + 目标市场）
- 检索结果 → 风险评分 → 预警生成 → EventBus 发布 → WS 推送

**边界定义（OUT scope）：**
- 不替换现有 `news_monitor`，仅扩展其数据源与分类能力
- 不修改 `risk_alert.py` 存储结构，通过 adapter 写入
- 不负责前端页面，仅提供 API

### 2.2 核心功能矩阵

| 功能 | 触发方式 | 执行路径 | 输出 |
|------|---------|----------|------|
| **主动关键词检索** | 用户 POST 请求 | `metaso_search` → LLM分析 → 风险评分 → 存储 | 结构化风险情报列表 |
| **周期自动检索** | APScheduler（可配 cron） | 关键词配置 → 批量检索 → 批量分析 → 预警生成 | 风险事件发布 + 预警写入 |
| **自动关键词推荐** | 产品新增事件 / 手动触发 | 产品HS编码 + 目标市场 → 关键词模板扩展 | 推荐关键词列表 |
| **历史情报检索** | 用户 GET 请求（带过滤） | SQLite 全文检索（FTS5） | 分页情报列表 |
| **风险热力图数据** | 定时聚合 | 三大域统计 + 时序趋势 | 仪表盘数据结构 |

### 3.2 数据库扩展方案

```sql
-- 表 1: 风险情报条目（扩展现有 monitored_news，不替换）
CREATE TABLE IF NOT EXISTS risk_intel_items (
    id           TEXT PRIMARY KEY,           -- MD5(source + title + pub_time)[:16]
    source_type  TEXT NOT NULL,              -- 'rss' | 'metaso' | 'webhook'
    source_name  TEXT NOT NULL,              -- 'reuters' | 'ft' | 'wsj' | ...
    title        TEXT NOT NULL,
    summary      TEXT,
    url          TEXT,
    pub_time     TEXT,                       -- ISO8601
    collected_at TEXT NOT NULL,              -- ISO8601
    -- 三级分类
    risk_domain  TEXT,                       -- tariff | conflict | financial | null
    risk_category TEXT,                      -- trade_war | interest_rate | ...
    -- 量化字段
    risk_score   REAL DEFAULT 0,             -- 0.0 ~ 1.0，LLM + 规则混合打分
    severity     TEXT DEFAULT 'low',         -- low | medium | high | critical
    sentiment    TEXT,                       -- positive | negative | neutral
    -- 市场关联
    affected_markets TEXT DEFAULT '[]',      -- JSON 数组: ["EU", "US", "CN"]
    affected_hs_codes TEXT DEFAULT '[]',     -- JSON 数组: ["8543", "9405"]
    -- 关键词溯源
    matched_keywords TEXT DEFAULT '[]',      -- 触发此条目的关键词 JSON 数组
    trigger_source   TEXT,                   -- 触发来源: 'auto' | 'user:xxx' | 'scheduled:xxx'
    -- 状态
    analyzed     INTEGER DEFAULT 0,          -- 0=未分析, 1=已分析
    alert_id     TEXT,                       -- 关联的 risk_alert id（若已触发预警）
    UNIQUE(source_name, title, pub_time)
);

CREATE INDEX IF NOT EXISTS idx_rii_domain ON risk_intel_items(risk_domain);
CREATE INDEX IF NOT EXISTS idx_rii_score  ON risk_intel_items(risk_score DESC);
CREATE INDEX IF NOT EXISTS idx_rii_time   ON risk_intel_items(collected_at DESC);
-- FTS5 全文索引（支持 title + summary 全文检索）
CREATE VIRTUAL TABLE IF NOT EXISTS risk_intel_fts USING fts5(
    id UNINDEXED, title, summary,
    content='risk_intel_items', content_rowid='rowid'
);

-- 表 2: 用户关键词配置（扩展现有 news_keyword_config，独立不覆盖）
CREATE TABLE IF NOT EXISTS risk_intel_keywords (
    id            TEXT PRIMARY KEY,          -- UUID
    user_id       TEXT NOT NULL,
    keyword       TEXT NOT NULL,
    label         TEXT,                      -- 用户自定义标签
    domain        TEXT,                      -- 预设关联域: tariff | conflict | financial | all
    auto_suggested INTEGER DEFAULT 0,        -- 1=系统推荐, 0=用户手动添加
    source_hint   TEXT,                      -- 触发推荐的来源: 'product:p001' | 'market:DE'
    -- 周期检索配置
    periodic_enabled INTEGER DEFAULT 0,      -- 是否开启周期检索
    cron_expr     TEXT DEFAULT '0 */6 * * *',-- cron 表达式，默认每 6 小时
    last_run_at   TEXT,                      -- 上次执行 ISO8601
    next_run_at   TEXT,                      -- 下次执行 ISO8601
    -- 统计
    total_runs    INTEGER DEFAULT 0,
    total_hits    INTEGER DEFAULT 0,         -- 累计触发预警次数
    created_at    TEXT NOT NULL,
    updated_at    TEXT NOT NULL,
    UNIQUE(user_id, keyword)
);

-- 表 3: 检索任务执行记录
CREATE TABLE IF NOT EXISTS risk_intel_runs (
    id            TEXT PRIMARY KEY,          -- UUID
    run_type      TEXT NOT NULL,             -- 'manual' | 'scheduled' | 'auto_suggest'
    keyword_id    TEXT,                      -- 关联 risk_intel_keywords.id（可空，全局扫描时为空）
    keyword       TEXT NOT NULL,             -- 快照关键词
    user_id       TEXT,                      -- 手动触发时的用户 ID
    status        TEXT DEFAULT 'pending',    -- pending | running | done | failed
    items_found   INTEGER DEFAULT 0,         -- 本次采集到的条目数
    items_new     INTEGER DEFAULT 0,         -- 去重后新增条目数
    alerts_created INTEGER DEFAULT 0,        -- 本次触发的预警数
    error_msg     TEXT,
    started_at    TEXT,
    finished_at   TEXT,
    created_at    TEXT NOT NULL
);
```

### 5.2 `RiskIntelEngine`（核心引擎）

```python
# app/core/risk_intel_engine.py

class RiskIntelEngine:
    """
    风险情报引擎 — 统一调度入口。
    
    职责：
    1. 关键词管理（增删改查推荐）
    2. 主动检索（metaso + RSS）
    3. 三级分类 + 风险评分
    4. 预警生成 + EventBus 发布
    5. 周期任务注册
    """

    # ── 关键词管理 ─────────────────────────────────
    
    def add_keyword(self, user_id: str, keyword: str, domain: str = "all",
                    periodic: bool = False, cron: str = "0 */6 * * *") -> dict
    """添加用户关键词；periodic=True 时同步注册 APScheduler 任务"""
    
    def remove_keyword(self, keyword_id: str, user_id: str) -> bool
    """删除关键词，同步注销对应调度任务"""
    
    def list_keywords(self, user_id: str, domain: Optional[str] = None) -> list[dict]
    """列出用户关键词，含统计字段（last_run_at/total_hits）"""
    
    def suggest_keywords(self, user_id: str, product_ids: Optional[list[str]] = None) -> list[dict]
    """
    基于用户产品的 HS 编码 + 目标市场，
    从 DOMAIN_KEYWORD_TEMPLATES 填充变量，返回推荐关键词列表。
    已存在的关键词自动去重。
    """

    # ── 主动检索 ───────────────────────────────────
    
    async def search(self, keyword: str, user_id: str,
                     domain: Optional[str] = None,
                     save: bool = True) -> SearchResult
    """
    实时联网检索（核心方法）。
    
    执行步骤：
    1. 调用 metaso_search(keyword, count=10) → 原始结果
    2. 并发调用 RSS 信源过滤（FTS 关键词匹配）
    3. 合并去重（基于 title 相似度）
    4. 调用 _classify_and_score() 批量打分
    5. save=True 时写入 risk_intel_items
    6. score > 0.7 时调用 _generate_alert()
    返回：SearchResult(items: list[RiskIntelItem], run_id: str)
    """
    
    async def batch_search(self, keyword_ids: list[str]) -> dict[str, SearchResult]
    """批量执行多个关键词检索，用于调度器周期任务"""

    # ── 分类与评分 ─────────────────────────────────
    
    def _classify_and_score(self, items: list[dict]) -> list[dict]
    """
    两阶段处理：
    
    阶段 1：规则引擎快速分类（无 LLM 调用）
      - 基于关键词字典（domain_keywords_dict）做 domain/category 初分
      - 基于信源权重（SOURCE_WEIGHTS）做 base_score
    
    阶段 2：LLM 精确分析（batched，≤20条/次）
      - 调用 risk_intel_analyzer.py
      - 输出：risk_domain, risk_category, risk_score(0-1), severity,
              affected_markets(list), affected_hs_codes(list), reasoning
    
    返回：enriched items 列表
    """

    # ── 预警生成 ───────────────────────────────────
    
    async def _generate_alert(self, item: dict, user_id: str) -> Optional[str]
    """
    条件：item.risk_score > 0.7 或 item.severity in ['high', 'critical']
    
    调用：risk_alert.create_alert()
    发布：EventBus.publish_raw({
        type: "risk:new_intel_alert",
        category: "risk_alert",
        data: {intel_id, alert_id, domain, category, score, title}
    })
    通知：ws_manager.broadcast({type: "alert", payload: {alert_id, ...}})
    返回：alert_id 或 None
    """

    # ── 周期任务管理 ───────────────────────────────
    
    def register_keyword_job(self, keyword_id: str, keyword: str, cron: str) -> str
    """在 APScheduler 注册周期检索任务，任务 ID = f"risk_intel_{keyword_id}" """
    
    def unregister_keyword_job(self, keyword_id: str) -> bool
    """注销调度任务"""
    
    async def run_all_periodic_keywords(self, user_id: Optional[str] = None) -> dict
    """全量执行所有 periodic_enabled=1 的关键词（定时任务入口）"""
```

## 十一、风险与约束

| 风险点 | 概率 | 影响 | 缓解措施 |
|--------|------|------|---------|
| 米塔搜索 API 限流 | 高 | 中 | 单用户每分钟最多 3 次检索；关键词去重+缓存（5min TTL） |
| RSS 信源不稳定 | 中 | 低 | 每个信源独立 try-except；失败仅跳过，不中断整体采集 |
| LLM 分析延迟 | 中 | 中 | 批处理（≤20条/次）；主动检索接口异步返回，分析结果通过 WS 推送 |
| FTS5 在旧版 SQLite 不可用 | 低 | 中 | 降级为 `LIKE` 全文搜索 |
| 周期任务过于密集（大量关键词） | 中 | 中 | 关键词数量上限：普通用户 20 个，admin 100 个；任务队列限流 |
| 三级分类误判 | 高 | 低 | 规则引擎兜底分类；LLM 仅负责精化，不覆盖规则引擎结果 |
