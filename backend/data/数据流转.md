## 二、L0 原始数据层（Raw Data）

### 2.1 HS 编码

| 属性 | 说明 |
|------|------|
| 位置 | `data/raw/hs_codes/` |
| 格式 | JSON |
| **使用者** | RuleEngine (`rule_engine.py`) |
| **使用条件** | 合规检查需要匹配产品到 HS 编码时 |
| **读取时机** | 模块加载时一次性读入内存缓存 |
| **更新方式** | 手动更新 JSON 文件 + 重启（或调用热加载 API） |
| **数据量** | 当前 15 条，目标 200+ 条 |

### 2.2 VAT 税率

| 属性 | 说明 |
|------|------|
| 位置 | `data/raw/vat_rates/` |
| 格式 | JSON |
| **使用者** | RuleEngine (`rule_engine.py`) |
| **使用条件** | 查询目标国家的 VAT 税率时 |
| **读取时机** | 模块加载时一次性读入内存缓存 |
| **更新方式** | 手动更新 JSON 文件 + 重启 |

### 2.3 认证矩阵

| 属性 | 说明 |
|------|------|
| 位置 | `data/raw/certifications/cert_matrix.json` |
| 格式 | JSON |
| **使用者** | RuleEngine → `get_certifications()` |
| **使用条件** | 查询目标国家所需产品认证时 |
| **读取时机** | 模块加载时一次性读入内存缓存（支持热加载） |
| **更新方式** | 修改 JSON 文件即生效（`config_watcher.py` 监听） |
| **说明** | 从 `rule_engine.py` 硬编码 `_cert_map` 迁移而来 |

### 2.4 法规知识（Markdown）

| 属性 | 说明 |
|------|------|
| 位置 | `data/raw/regulations/` |
| 格式 | Markdown (.md) |
| **使用者** | Knowledge Loader → ChromaDB (L1) |
| **使用条件** | 知识库初始化/更新时，先读原始 Markdown 再分块向量化 |
| **读取时机** | `init_knowledge.py` 执行时；定时刷新时。优先 `data/raw/regulations/{market}/_all.md`，回退到 `data/regulations.md` |
| **更新方式** | 修改 Markdown 文件后重新执行向量化（`init_knowledge.py --market {market}`） |

## 三、L1 知识库层（Knowledge Base）

### 3.1 ChromaDB 向量库

| 属性 | 说明 |
|------|------|
| 位置 | `data/knowledge/chroma/{market}/` |
| 格式 | ChromaDB 持久化目录（按市场分 collection） |
| 市场隔离 | `eu/`, `us/`, `jp/`, `kr/` 各自独立 collection，store.py 自动路由：检测 query 关键词（如"欧盟"→eu、"FCC"→us）|
| **使用者** | RAG (`rag.py`) |
| **使用条件** | 规则引擎无法覆盖时，检索相关法规知识辅助回答 |
| **读取时机** | 每次合规查询都可能触发检索（语义相似度搜索） |
| **写入时机** | 知识库初始化/增量更新时 |
| **写入者** | `init_knowledge.py` / scheduler 定时任务 |
| **降级策略** | 检索失败/Chromab 不可用 → 返回空结果，不阻塞主流程 |

### 3.2 检索触发条件

| 场景 | 触发条件 | 使用数据 |
|------|----------|---------|
| 确定性合规检查 | 用户明确指定产品+国家 | L0 + RuleEngine（不经过 L1） |
| 开放式法规咨询 | "欧盟对电子产品有什么要求" | L1 ChromaDB + RAG + LLM |
| 验证性查询 | 规则引擎结果需要法规来源引用 | L1 ChromaDB + RAG |
| LLM 兜底 | 规则引擎无法匹配任何已知品类 | L1 + L3 + L4 + LLM |

## 四、L2 项目/产品记忆层（Project Memory）

### 4.1 产品合规档案

| 属性 | 说明 |
|------|------|
| 位置 | `data/project_memory/{product_id}/` |
| 格式 | JSON（基于 NLRecord 模型） |
| **使用者** | 历史查询 · Dashboard · 合规报告页面 |
| **使用条件** | 查看产品历史合规记录时 |
| **读取时机** | 用户查询某产品历史的合规结果时 |
| **写入时机** | 每次合规检查完成后写入 |
| **写入者** | `compliance.py` / `chat.py` |

### 4.2 存储内容

```json
{
  "product_id": "p_xxx",
  "product_name": "LED灯带",
  "target_market": "德国",
  "last_check": "2026-05-24T10:30:00Z",
  "history": [
    {
      "check_id": "chk_xxx",
      "timestamp": "2026-05-24T10:30:00Z",
      "result": { "hs_code": "9405.40", "vat_rate": 19.0, ... }
    }
  ]
}
```

## 五、L3 用户记忆层（User Memory）

### 5.1 用户画像/偏好

| 属性 | 说明 |
|------|------|
| 位置 | `data/user_memory/{user_id}/` |
| 格式 | JSON（基于 NLRecord 模型） |
| **使用者** | NLU · LLM 兜底 · 个性化回复 |
| **使用条件** | LLM 推理需要用户上下文时 |
| **读取时机** | 每次用户消息到达时加载 |
| **写入时机** | 用户偏好明确时更新（如常用目标市场） |
| **写入者** | LLM 推理后提取 / 用户手动设置 |

### 5.2 存储内容

```json
{
  "user_id": "u_xxx",
  "preferred_markets": ["德国", "法国"],
  "recent_searches": ["电子产品", "LED灯带"],
  "company_info": { "name": "XX贸易公司", "products": [] }
}
```

### 5.3 使用场景

| 场景 | 读取 | 写入 |
|------|------|------|
| 用户打开对话框 | L3（偏好）+ L4（上轮上下文） | — |
| NLU 解析意图 | L3（常用市场辅助消歧） | — |
| LLM 生成回复 | L3（个性化） | — |
| 用户修改偏好 | — | L3 |
| 合规检查完成 | — | L2 + L5 |

### 6.1 会话上下文

| 属性 | 说明 |
|------|------|
| 位置 | `data/session_memory/{user_id}/sessions/` |
| 格式 | JSON（基于 NLRecord 模型，不做 TTL） |
| **使用者** | NLU · LLM 兜底 · 对话连续性 |
| **使用条件** | 需要维持多轮对话上下文时 |
| **读取时机** | 每次用户消息到达时 |
| **写入时机** | 每轮问答结束后 |
| **写入者** | `chat.py` / `compliance.py` |

### 6.2 存储内容

```json
{
  "session_id": "sess_xxx",
  "user_id": "u_xxx",
  "messages": [
    {"role": "user", "content": "LED灯带出口德国需要什么认证"},
    {"role": "assistant", "content": "...", "compliance_result": {...}}
  ],
  "current_product": "LED灯带",
  "current_market": "德国"
}
```

## 七、L5 事件链层（Event Chain）

### 7.1 系统事件 + 操作链（合并）

| 属性 | 说明 |
|------|------|
| 位置 | `data/event_chain/system_events/` + `data/event_chain/action_chains/` |
| 格式 | JSON |
| **使用者** | 回溯排查 · Dashboard 事件时间线 · 审计 |
| **使用条件** | 审计追踪 / 事件监控 / 决策回溯 |
| **读取时机** | 用户查看事件时间线/决策链路时 |
| **写入时机** | 每次操作/事件发生时 |
| **写入者** | `chat.py` · `rule_engine.py` · shopify webhook · scheduler 任务 |

### 7.2 事件类型说明

| 事件类型 | 触发者 | 读取数据 | 写入数据 |
|----------|--------|----------|----------|
| `user_query` | 用户发送消息 | L3 + L4 | L5 action_chain |
| `rule_check` | RuleEngine 执行 | L0 | L5 action_chain |
| `rag_retrieve` | RAG 检索 | L1 | L5 action_chain |
| `compliance_report` | 合规检查完成 | L0+L1+L2 | L2 + L5 |
| `shopify_import` | Shopify 导入 | Shop API | L2 + L5 |
| `shopify_webhook` | 产品变更通知 | Shop Webhook | L5 |
| `regulation_update` | 法规变更（scheduler） | 外部 API | L1 + L5 |
| `cert_expiry_alert` | 认证到期提醒 | L2 | L5 |

## 八、组件数据使用矩阵

| 组件 | 读取层 | 写入层 | 触发条件 |
|------|--------|--------|----------|
| **NLU (LLM)** | L3(偏好), L4(上下文) | L5(action_chain) | 用户消息 → 意图+实体解析 |
| **RuleEngine** | L0(hs_codes/vat/cert) | L5(action_chain) | 明确产品+国家 → 确定性检查 |
| **RAG** | L1(ChromaDB) | L5(action_chain) | 规则引擎无法覆盖 / 开放问答 |
| **LLM 兜底** | L1, L3, L4 | L5(action_chain) | 规则引擎无匹配 → 智能推断 |
| **合规报告** | L0+L1 | L2(project), L5(event) | 所有检查完成后 |
| **Shopify 集成** | Shopify API | L2, L5 | 用户授权/导入/check |
| **Scheduler** | L0+L1 | L1, L5 | 定时触发 |
| **用户设置** | — | L3 | 用户主动修改 |
| **Codex Agent** | 外部网络（联网搜索） | Codex 结构化结果 | Scheduler 定时 / API 手动 |
| **Market Monitor** | L2(产品列表), Codex 结果 | L5(market_event) | 市场扫描触发 |
| **Risk Alert Engine** | L0(市场事件) | data/risk_alerts/ | 检测到重要变更 |
| **WebSocket Manager** | — | WebSocket 连接 | 用户连接 / alert 生成 |
| **Metrics** | L2+L3+L5 | —（只读聚合） | 定时 / API 请求 |

## 九、数据更新流程

### 9.1 合规查询完整数据流

```
用户输入
  │
  ├─→ [读取] L3 user_memory(偏好) + L4 session(上下文)
  │
  ├─→ [NLU·LLM] 解析意图
  │     └─→ [写入] L5 action_chain(操作记录)
  │
  ├─→ [RuleEngine] 确定性检查
  │     ├─ [读取] L0 hs_codes / vat_rates / cert_matrix
  │     └─ [写入] L5 action_chain
  │
  ├─→ [RAG] 知识库检索（规则引擎不够时）
  │     ├─ [读取] L1 ChromaDB
  │     └─ [写入] L5 action_chain
  │
  ├─→ [LLM Fallback]（规则引擎无匹配时）
  │     ├─ [读取] L1 + L3 + L4
  │     └─ [写入] L5 action_chain
  │
  └─→ [组装回复]
        ├─ [写入] L2 project_memory(产品合规档案)
        ├─ [写入] L4 session_memory(会话记录)
        └─ [写入] L5 event(操作链)
```

### 9.2 数据更新事件流

```
定时器触发 / Webhook 接收
  │
  └─→ [Scheduler] / [Shopify Webhook]
        ├─ [读取] 外部数据源 / L0 现有数据
        ├─ [写入] L1(知识库增量更新)
        ├─ [写入] L5(事件记录)
        └─ [可选] 触发合规检查 → L2 更新
```

### 9.3 风险监控与预警数据流

```
[Scheduler 定时 / API 手动]
  │
  ├─→ [Codex Agent] 联网搜索 EU/US/JP/KR 市场法规变更
  │     ├─ 使用 prompt: market_monitor.yaml
  │     ├─ Codex 自主搜索 + 浏览 + 分析
  │     └─ 返回结构化市场事件 JSON
  │
  ├─→ [Market Monitor] 解析 Codex 结果
  │     └─→ [写入] L5 event_store (market_event)
  │
  ├─→ [Codex Agent] 影响分析
  │     ├─ [读取] L2 project_memory (用户产品列表)
  │     ├─ 使用 prompt: impact_analysis.yaml
  │     └─ 返回受影响的 products + impact 级别
  │
  ├─→ [Risk Alert Engine]
  │     ├─ 生成 RiskAlert (类型/严重度/标题/描述)
  │     └─ [写入] data/risk_alerts/{user_id}/alerts.json
  │
  ├─→ [WebSocket Manager] 实时推送
  │     └─→ 前端 AlertPanel 更新未读数 + 列表
  │
  └─→ [API 轮询]
        └─→ GET /api/v1/risk/alerts (前端备选拉取方式)
```

### 9.4 Prompt 模板使用矩阵

| Prompt 文件 | 使用者 | 场景 | 可微调策略 |
|------------|--------|------|-----------|
| `market_monitor.yaml` | Codex Agent | 市场法规变更联网搜索 | 调整搜索源/关键词/输出格式 |
| `impact_analysis.yaml` | Codex Agent | 法规变更对用户产品的影响 | 调整影响评估标准 |
| `risk_summary.yaml` | Codex Agent | 风险摘要生成 | 调整摘要风格/长度 |
| `nlu_fallback.yaml` | NLU (`nlu.py`) | LLM 意图解析兜底 | 调整意图分类/字段映射 |

调用 `POST /api/v1/prompts/reload` 可热加载所有修改后的 prompt 模板。
