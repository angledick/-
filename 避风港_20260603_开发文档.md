## 一、系统架构全景

### 1.1 设计哲学

| 原则 | 说明 |
|------|------|
| 事件驱动，主动感知 | 不等用户提问，系统主动感知合规热点并推送 |
| 记忆优先，上下文完整 | 所有数据规范化为记忆树，Agent每天已拥有全天上下文 |
| 多Agent协作，职责隔离 | Manager拆解任务，Worker隔离执行，群聊式调度 |
| Token经济，成本可控 | 所有工具输出经TokenJuice压缩，降低80%token消耗 |
| 本地优先，数据主权 | 记忆/配置/密钥存储在本地，加密不动 |
| OS级组件化 | 像操作系统一样提供进程管理、文件系统、权限控制、网络栈 |

## 三、重要功能详解

### 3.1 事件驱动六阶段合规流水线

```
感知(Sense) → 检查(Check) → 推荐(Recommend) → 告知(Notify) → 交互(Interact) → 处理(Process)
```

| 阶段 | 执行者 | 输入 | 输出 |
|------|--------|------|------|
| 感知 | EventBus + MarketMonitor | 用户动作/外部API/定时调度/规则引擎 | 标准化EventRecord |
| 检查 | Worker + Skill + RuleEngine | EventRecord | 检查结果 + 风险等级 |
| 推荐 | Skill推荐器 + 规则引擎 | 检查结果 | 含置信度的操作建议列表(RecommendAction) |
| 告知 | 通知引擎 + 频道适配器 | 推荐结果 | 多端消息推送(Dashboard/飞书/钉钉) |
| 交互 | SSE对话引擎 + Cowork | 用户响应 | 执行指令(确认/修改/委派) |
| 处理 | Workflow + Skill流 | 确认指令 | 操作结果 + MCP回写 |

### 3.2 多Agent调度（聊天室模式）

```
Manager Agent (协调者)
  │  任务拆解 + 分配 + 监督
  │  读取全局记忆：活跃Agent列表 + 各Agent事件流
  ├── compliance_worker  — 合规检查（全阶段）
  ├── certification_worker — 认证管理（全阶段）
  ├── store_worker — 建站配置（阶段1）
  ├── product_worker — 选品设计（阶段2）
  ├── supplier_worker — 供应商审核（阶段3）
  ├── listing_worker — 商品上架（阶段4）
  ├── payment_worker — 支付配置（阶段5）
  ├── order_worker — 订单处理（阶段6）
  ├── customs_worker — 出口报关（阶段7）
  ├── logistics_worker — 物流配送（阶段8）
  ├── aftersale_worker — 售后退货（阶段9）
  ├── finance_worker — 财务结算（阶段10）
  └── monitoring_worker — 指标监控（全阶段）
```

### 3.3 产品级隔离存储

```
data/products/{product_id}/
├── events/          # 产品专属事件链（chain.json）
├── metrics/         # 产品专属指标池（metrics.json + history.json 90天）
├── memory/          # 产品专属记忆库（compliance/issues/notes/metadata）
└── knowledge/       # 产品专属知识库（regulations.md + certifications.md + chroma/）

data/global/
├── events/          # 全局事件总线（bus.json + archive/按月归档）
├── knowledge/       # 全局知识库（ChromaDB向量索引）
├── memory/          # 全局共享记忆（活跃市场/品类/法规变更）
└── metrics/         # 全局聚合指标（健康度/高风险占比/认证到期分布）
```

### 3.4 SSE流式对话协议

```typescript
type StreamEvent =
  | { type: 'token'; content: string }                     // AI回复逐token输出
  | { type: 'skill_start'; skill: string; args: object }   // Skill开始执行
  | { type: 'skill_end'; skill: string; result: object; duration_ms?: number; status: 'success'|'error' }
  | { type: 'thinking'; content: string; depth?: number }  // Agent思考过程
  | { type: 'plan'; steps: PlanStep[]; current: number }   // 执行计划+当前步骤
  | { type: 'action_card'; actions: Action[] }             // 操作建议卡片
  | { type: 'error'; code: string; message: string; recoverable?: boolean }
  | { type: 'done'; finish_reason?: string; usage?: object }
```

### 3.5 记忆树（4层级摘要）

```
L3 全局索引（≤500 token/产品） — 所有领域概览的索引
  └── L2 领域概览（≤1k token） — 德国市场合规全景 / LED产品线全景
       └── L1 话题摘要（≤2k token） — WEEE认证到期 / CE标志检查
            └── L0 原始片段（≤3k token） — 合规记录 / 事件条目 / 人工备注
```

存储：SQLite（fragments + summaries两表） + Obsidian Wiki(.md文件兼容双链)

### 3.6 四大支柱协作模型

```
MCP（数据层）── 事件感知 + 结果回写
  ↓
Skill（规则层）── 合规检查 + 操作推荐
  ↓
Workflow（确定性层）── 步骤编排 + 状态追踪 + 回滚
  ↑
Cowork（入口层）── SSE对话 + 用户确认 + 操作卡片
```

## 五、事件流转

### 5.1 8类事件体系

| 事件类别 | 编码 | 作用域 | 子类示例 |
|----------|------|--------|----------|
| 生命周期事件 | `lifecycle` | 产品级 | product:created / status_changed / ended |
| 合规检查事件 | `compliance` | 产品级 | check_started / check_passed / check_failed |
| 认证管理事件 | `certification` | 产品级 | uploaded / expiring / expired |
| 订单事件 | `order` | 产品级 | created / shipped / returned |
| 市场法规事件 | `regulation` | 全局 | updated / new / repealed |
| 风险预警事件 | `risk_alert` | 全局/产品 | new / resolved / escalated |
| 系统事件 | `system` | 全局 | config_changed / error / health_check |
| 用户操作事件 | `user_action` | 全局 | login / config_modified / approved |

### 5.2 事件订阅机制（4种模式）

| 订阅类型 | 说明 | 适用场景 |
|----------|------|----------|
| 精准订阅 | 特定产品+特定事件类型 | 单产品合规结果监听 |
| 批量订阅 | 一组产品的特定事件 | 批次产品认证到期监控 |
| 全局订阅 | 全部产品的某类事件 | 全局法规变更通知 |
| 条件订阅 | 按字段条件过滤(severity=critical) | 仅高风险事件告警 |

## 六、业务流程

### 6.1 跨境合规10大业务阶段

| 阶段 | 业务环节 | 核心能力 | 对应Worker | 产品状态机 |
|------|----------|----------|-----------|-----------|
| 1 | 建站与基础环境搭建 | 第三方集成+QAAgent | store_worker | concept |
| 2 | 选品与样品设计 | 记忆树+RAG | product_worker | design |
| 3 | 供应商审核与采购 | 事件驱动+多Agent | supplier_worker | sourcing |
| 4 | 商品上架与内容合规 | 规则引擎+TokenJuice | listing_worker | ready |
| 5 | 支付与收款配置 | 第三方集成+安全沙箱 | payment_worker | ready |
| 6 | 订单处理与境内物流 | 事件驱动+主动引擎 | order_worker | active→fulfilling |
| 7 | 出口报关（跨境干线）| 多Agent+记忆树 | customs_worker | fulfilling |

### 6.2 产品生命周期状态机（8阶段）

```
concept → design → sourcing → ready → active → fulfilling → aftersale → end
                                         ↑          │            │
                                         └──────────┘            │
                                         ↑                       │
                                         └───────────────────────┘
```

合法转换：支持active↔fulfilling、fulfilling→aftersale、aftersale→active/end的回退

### 6.3 六步执行流水线（对齐指南§6.15）

```
Step 1: MCP感知 (perceive)
  → 数据源接入：Shopify Webhook / MarketMonitor / 用户操作
  → 输出：标准化EventRecord

Step 2: 规则检查 (check)
  → RuleEngine规则匹配 + RAG检索相关法规
  → 输出：CheckResult(风险等级 + 检查详情)

Step 3: 推荐操作 (recommend) ★新增独立步骤
  → Skill推荐器 + 规则库匹配
  → 输出：RecommendAction[]{action, confidence, skill, expected_result, risk_level}

Step 4: 用户通知 (notify)
  → Cowork多渠道告知：Dashboard弹窗/飞书/钉钉/邮件/Webhook
  → 通知payload必含：product_id + stage（深度链接跳转）

Step 5: 对话交互 (interact)
  → SSE流式对话：展示推荐→用户确认/修正→生成执行指令

Step 6: Workflow执行 (execute)
  → 确定性Workflow编排 + Skill流执行 + MCP结果回写
  → 输出：操作结果 + 事件链追加 + 指标更新
```

### 6.4 权限模型与审批流

**5角色体系**：super_admin / ops_admin / compliance_officer / product_manager / viewer

**审批流规则**：
- 批量上架 → 合规审查员24h内审批
- 高风险品操作 → 双审批，72h超时自动驳回
- 认证豁免 → 超管48h内审批

### 6.5 通知体系

| 触发条件 | 优先级 | 通道 |
|----------|--------|------|
| 认证30天到期 | 高 | Dashboard + 飞书/钉钉 |
| 认证已过期 | 紧急 | Dashboard + 飞书 + 邮件 |
| 法规变更 | 高 | Dashboard + 飞书/钉钉 |
| 合规检查失败 | 中 | Dashboard + WebSocket |
| 产品状态变更 | 低 | Dashboard |
| 拒付率超标 | 高 | Dashboard + 飞书 + 邮件 |
