# 前端变更路线图 · 体验重设计（合并版）

## 核心设计理念转变

**现有问题**：
1. 无流式对话能力，用户无法完成实时交互
2. 无工具/技能/Agent配置切换界面，用户无法通过对话配置对接系统
3. 业务合规流未区分系统级与产品级
4. 对话不展示Skill事件/思考过程/执行计划，用户看不到AI在做什么

**新设计原则**：
1. **流式对话是一切的基础** — SSE实时推送token + skill事件 + 思考过程 + 执行计划，所有业务操作最终通过对话完成
2. **系统合规 vs 产品合规双层架构** — 系统合规：全局展示；产品合规：产品卡片，点开即对话
3. **工具/技能/Agent可配置可切换** — 用户可选择工具集、技能、Agent，通过对话配置对接系统
4. **对话即操作台** — 对话不只是问答，而是展示Skill调用、思考链、Workflow步骤、操作建议的完整操作台
5. **事件驱动的主动体验** — 系统主动推送合规事件和操作建议，用户确认即执行

## 流式对话核心设计

### SSE消息协议

对话通过SSE（Server-Sent Events）流式推送，每条消息携带类型标识：

```typescript
// SSE消息类型定义
type StreamEvent =
  | { type: 'token'; content: string }                    // AI回复token逐字输出
  | { type: 'skill_start'; skill: string; args: object }  // Skill开始执行
  | { type: 'skill_end'; skill: string; result: object; duration_ms?: number; status?: 'success' | 'error' }  // Skill执行完成（扩展字段可选）
  | { type: 'thinking'; content: string; depth?: number } // Agent思考过程（depth为扩展字段）
  | { type: 'plan'; steps: PlanStep[]; current: number }   // 执行计划+当前步骤
  | { type: 'action_card'; actions: Action[] }             // 操作建议卡片
  | { type: 'error'; code: string; message: string; recoverable?: boolean }  // 错误事件
  | { type: 'done'; finish_reason?: string; usage?: object }  // 对话轮次结束（扩展字段可选）
```

> **字段说明**：后端§6.9.3 中标注 `★` 的为前端必须消费字段，其余为扩展字段（前端可忽略或透传）。
> 扩展字段使用策略：前端渲染时直接透传至组件props，若组件未定义对应prop则静默忽略。

### SSE连接管理

```typescript
// SSE连接配置与重连策略
interface SSEConnectionConfig {
  url: string;                          // SSE端点，如 /api/v1/chat/stream
  method?: 'GET' | 'POST';              // 默认为GET，支持POST（通过fetch + ReadableStream）
  onMessage: (event: StreamEvent) => void;
  onError?: (error: SSEError) => void;
  onStatusChange?: (status: ConnectionStatus) => void;
}

type ConnectionStatus = 'connecting' | 'connected' | 'reconnecting' | 'disconnected' | 'error';

interface SSEError {
  code: string;                         // 错误码
  message: string;                      // 错误描述
  recoverable: boolean;                 // 是否可自动恢复
  retryAttempt?: number;                // 当前重试次数
}

// 自动重连策略（指数退避）
const RECONNECT_CONFIG = {
  maxRetries: 5,                        // 最大重试次数
  initialDelay: 1000,                   // 初始延迟1秒
  maxDelay: 30000,                      // 最大延迟30秒
  backoffFactor: 2,                     // 退避因子
  jitter: 0.1,                          // 随机抖动0.1
};

// 超时处理
const TIMEOUT_CONFIG = {
  connectionTimeout: 15000,             // 连接超时15秒
  idleTimeout: 60000,                   // 空闲超时60秒（无任何事件）
  heartbeatInterval: 30000,             // 心跳间隔30秒
};
```

### 流式对话组件渲染规则

```typescript
// 消息区渲染：根据type渲染不同UI块
const StreamMessageRenderer: React.FC<{ events: StreamEvent[] }> = ({ events }) => {
  return (
    <div className="stream-container">
      {events.map((event, i) => {
        switch (event.type) {
          case 'token':
            return <span key={i} className="ai-token">{event.content}</span>
          case 'skill_start':
            return <SkillEventBlock key={i} status="running" skill={event.skill} args={event.args} />
          case 'skill_end':
            return <SkillEventBlock key={i} status="done" skill={event.skill} result={event.result} />
          case 'thinking':
            return <ThinkingBlock key={i} content={event.content} collapsible />
          case 'plan':
            return <PlanBlock key={i} steps={event.steps} current={event.current} />
          case 'action_card':
            return <ActionCardGroup key={i} actions={event.actions} />
          case 'error':
            return <ErrorMessageBlock key={i} code={event.code} message={event.message} recoverable={event.recoverable} />
        }
      })}
    </div>
  )
}
```

### 文件导入流程

```
用户操作 → 系统处理:

1. GitHub链接导入:
   用户输入: https://github.com/org/skill-name
   系统: SDK解析仓库 → 检测skill.yaml → 生成配置 → 安装

2. ZIP文件上传:
   用户上传: skill-package.zip
   系统: 解压 → 检测配置文件 → 验证格式 → 安装

3. 手动撰写MD:
   用户编写: skill-config.md (含YAML frontmatter)
   系统: 解析MD → 提取配置 → 生成Skill → 安装

4. 文件刷新:
   用户点击: [刷新文件变更]
   系统: 重新读取源文件 → 对比差异 → 更新配置
```

### Agent配置联动设计

```typescript
// 权限类型定义
interface Permission {
  resource: string;                    // 资源标识，如 'product:read', 'skill:write', 'config:admin'
  actions: ('create' | 'read' | 'update' | 'delete' | 'execute')[];
  scope?: 'self' | 'team' | 'all';    // 作用域
  conditions?: Record<string, any>;    // 条件约束，如 { stage: ['concept', 'design'] }
}

// 事件类型定义（对齐后端§6.14 EventType接口）
interface EventType {
  category: 'lifecycle' | 'compliance' | 'certification' | 'order' | 'regulation' | 'risk_alert' | 'system' | 'user_action';
  name: string;                        // 事件名称，如 'compliance:check_failed'
  description?: string;                // 事件描述
  scope: 'product' | 'global';        // 作用域
  data_sources?: {                     // 数据血缘
    read: string[];                    // 读取的数据源
    write: string[];                   // 写入的数据源
  };
  config_file?: string;               // 对应事件配置文件路径
  schema?: Record<string, any>;        // 事件数据Schema
}

// Agent配置（扩展category字段）
interface AgentConfig {
  id: string;
  name: string;
  role: 'manager' | 'worker' | 'qa';
  category?: string;                   // 仅Worker类型需要，如 'lifecycle' | 'compliance' | 'certification'
  skills: string[];                    // 关联的Skill ID列表
  tools: string[];                     // 关联的Tool ID列表
  oauth: string[];                     // 关联的OAuth ID列表
  permissions: Permission[];
}

// 操作建议类型
interface Action {
  id: string;
  label: string;                       // 操作显示名称，如 "续期WEEE认证"
  description?: string;                // 操作说明
  skill?: string;                      // 执行Skill
  confidence?: number;                 // AI置信度 0.0~1.0
  expected_result?: string;            // 预期结果
  risk_level?: 'low' | 'medium' | 'high';
  status?: 'pending' | 'confirmed' | 'executing' | 'done' | 'skipped';
}

// 执行计划步骤类型
interface PlanStep {
  id: string;
  action: string;                      // 步骤描述
  expected_result?: string;            // 预期结果
  status: 'pending' | 'running' | 'done' | 'failed';
  skill?: string;                      // 执行Skill
  duration_ms?: number;                // 执行耗时
}

// 后端API联动
const configAPIs = {
  // Agent CRUD
  getAgents: () => GET('/api/v1/agents'),
  createAgent: (data) => POST('/api/v1/agents', data),
  updateAgent: (id, data) => PUT(`/api/v1/agents/${id}`, data),
  deleteAgent: (id) => DELETE(`/api/v1/agents/${id}`),
  
  // Agent关联配置
  getAgentSkills: (agentId) => GET(`/api/v1/agents/${agentId}/skills`),
  setAgentSkills: (agentId, skillIds) => PUT(`/api/v1/agents/${agentId}/skills`, { skillIds }),
  getAgentTools: (agentId) => GET(`/api/v1/agents/${agentId}/tools`),
  setAgentTools: (agentId, toolIds) => PUT(`/api/v1/agents/${agentId}/tools`, { toolIds }),
  getAgentOAuth: (agentId) => GET(`/api/v1/agents/${agentId}/oauth`),
  setAgentOAuth: (agentId, oauthIds) => PUT(`/api/v1/agents/${agentId}/oauth`, { oauthIds }),
  
  // Skills CRUD
  getSkills: () => GET('/api/v1/skills'),
  installSkill: (source) => POST('/api/v1/skills/install', source),  // source: github/zip/md
  updateSkill: (id) => PUT(`/api/v1/skills/${id}/config`),  // 使用PUT /config 避免与POST /update功能重叠（对齐后端§6.13）
  deleteSkill: (id) => DELETE(`/api/v1/skills/${id}`),
  refreshSkill: (id) => POST(`/api/v1/skills/${id}/refresh`),
  
  // Tools CRUD
  getTools: () => GET('/api/v1/tools'),
  createTool: (data) => POST('/api/v1/tools', data),
  updateTool: (id, data) => PUT(`/api/v1/tools/${id}`, data),
  deleteTool: (id) => DELETE(`/api/v1/tools/${id}`),
  
  // OAuth CRUD（实际路由指向 integrations/，oauth/ 仅为兼容别名）
  getOAuth: () => GET('/api/v1/integrations'),
  createOAuth: (data) => POST('/api/v1/integrations', data),
  updateOAuth: (id, data) => PUT(`/api/v1/integrations/${id}`, data),
  deleteOAuth: (id) => DELETE(`/api/v1/integrations/${id}`),
  testOAuth: (id) => POST(`/api/v1/oauth/${id}/test`),  // 保留OAuth test快捷入口
```

## 分阶段实施路线

### Phase 1: 流式对话核心 + 系统合规 + 产品卡片 + 配置中心 + 布局骨架（10-14周）

**目标**：建立流式对话基础能力，实现系统合规全局展示、产品合规卡片入口、配置中心管理，用户可完成基本的流式对话交互和配置管理。

> **前后端同步点**：以下标注了前端各模块依赖的后端API所在的Phase。
> 前端Phase 1大部分功能依赖后端Phase 2/3 API，建议：
> 1. SSE流式对话使用Mock SSE Server独立开发（见下方Mock策略），待后端Phase 2 SSE API就绪后切换
> 2. 产品管理API（后端Phase 1已交付）可正常并行开发
> 3. 配置中心Skills管理界面需后端Phase 3 Skills API，Phase 1先用Mock数据填充UI

**前后端Phase依存关系表：**

| 前端模块 | 依赖后端API | 后端Phase | 前端开发策略 |
|----------|------------|-----------|------------|
| SSE流式对话引擎 | `/api/v1/chat/stream` | P2 | 先用Mock SSE Server开发 |
| Agent/工具/技能选择器 | `/api/v1/agents`, `/tools`, `/skills` | P1(agents) / P3(skills) | agents用真实API，skills先Mock |
| 配置中心-Skills管理 | `/api/v1/skills/*` | P3 | Mock数据填充UI |
| 配置中心-Tools管理 | `/api/v1/tools/*` | P2 | 先Mock后切换（后端P2才实现Tools CRUD） |
| 配置中心-OAuth管理 | `/api/v1/oauth/*`/`/api/v1/integrations/*` | P2 | 先Mock后切换 |
| 产品卡片网格 | `/api/v1/products` | P1 | ✅ 真实API，可并行 |
| 产品对话界面 | `/api/v1/products/:id`, `/api/v1/chat/stream` | P1(product API) / P2(SSE) | 先用Mock SSE |
| 系统合规页面 | `/api/v1/pipeline/stages` | P2 | Mock数据填充 |

**Mock SSE Server策略：**

建议在 Phase 1 初始阶段构建独立的 Mock SSE Server，用于前端SSE流式对话的独立开发与测试：

```
# Mock SSE Server 方案
# 方案A：在 frontend/ 目录下搭建简易Node.js脚本（开发依赖）
frontend/mock/
  ├── sse-server.js          # 基于Node.js + Express的Mock SSE端点
  └── mock-data.js           # 模拟token/skill/thinking/plan等事件流

# 方案B：在后端代码中以条件编译方式添加Mock端点（仅开发环境启用）
backend/app/api/mock_sse.py  # 返回硬编码的SSE事件流
```

Mock SSE Server 支持的事件流：token逐字输出、skill_start/skill_end事件、thinking过程、plan步骤、action_card建议。
前端通过环境变量 `VITE_SSE_ENDPOINT` 切换真实/ Mock 端点。

### Phase 2: 主动告知 + 操作执行闭环 + 事件驱动

> **拆分建议**：前端Phase 2范围（通知UI+操作闭环+产品详情增强）远小于后端Phase 2范围（记忆树+多Agent+主动引擎+SSE+Workflow+Pipeline+Tools+指标+洞察），

## 前后端联动点

### 数据流

| 用户场景 | 后端提供 | 前端展示 | 联动方式 |
|----------|----------|----------|----------|
| **流式对话** | SSE `/api/v1/chat/stream` | 逐token输出 + Skill事件 + 思考 + 计划 | SSE |
| **Agent/工具/技能列表** | GET `/api/v1/agents` `/tools` `/skills` | 选择器/面板 | REST API |
| **对话配置** | GET/PUT `/api/v1/chat/config` | Agent/工具/技能切换 | REST API |
| **Agent配置CRUD** | REST `/api/v1/agents/*` | Agent卡片列表 | REST API |
| **Skills管理** | REST `/api/v1/skills/*` | Skill卡片+导入 | REST API |
| **Tools管理** | REST `/api/v1/tools/*` | Tool卡片 | REST API |
| **OAuth管理** | REST `/api/v1/integrations/*`（oauth/为兼容别名） | OAuth卡片+配置 | REST API |
| **系统合规状态** | GET `/api/v1/pipeline/stages` | 10阶段时间轴 | REST API |
| **产品列表** | GET `/api/v1/products` | 产品卡片网格 | REST API |
| **产品详情** | GET `/api/v1/products/:id` | 产品信息面板 | REST API |
| **产品事件** | GET `/api/v1/products/:id/events` | 事件时间线 | REST + WS |
| **产品指标** | GET `/api/v1/products/:id/metrics` | 指标看板 | REST API |
| **CLI命令执行** | POST `/api/v1/cli/execute` | CLI命令结果展示 | REST API |
| **魔法命令执行** | POST `/api/v1/cli/magic` | 对话快捷命令结果 | REST API |
| **CLI命令自动补全** | GET `/api/v1/cli/complete` | 命令自动补全建议 | REST API |
| **CLI命令历史** | GET `/api/v1/cli/history` | 命令执行历史列表 | REST API |

### 事件推送 (WebSocket)

| 事件类型（蛇形命名，对齐后端§6.9.5 NotificationPayload.type） | 前端响应 |
|----------|----------|
| `product_created` | 产品卡片网格刷新 |
| `compliance_check_failed` | 通知弹窗，点击进入产品对话 |
| `certification_expiry` | 高优先级通知 + 推荐操作 |
| `regulation_change` | 概览页简报更新 |
| `agent_task_update` | 对话区执行计划块更新 |
| `config_skill_installed` | Skills列表刷新 |
| `config_tool_updated` | Tools列表刷新 |
| `config_oauth_connected` | OAuth状态更新 |

## 验收标准

### Phase 1 验收
- [ ] 流式对话：SSE连接正常，AI回复逐token输出，无闪烁
- [ ] Skill事件块：展示Skill调用状态（运行中/完成/失败）、入参、结果、耗时
- [ ] 思考过程块：可折叠展示Agent推理过程
- [ ] 执行计划块：展示步骤列表、当前进度、步骤状态
- [ ] 操作建议卡片：可交互（确认/修改/执行/跳过）
- [ ] Agent选择器：可切换不同Agent
- [ ] 工具面板：显示已启用工具列表，可开关
- [ ] 技能面板：显示已安装Skills，可激活/停用
- [ ] 系统合规页面：10阶段全局合规总览，点击展开阶段详情
- [ ] 产品合规页面：产品卡片网格，按生命周期分组
- [ ] 产品对话界面：点击产品卡片进入三栏对话界面，流式对话正常
- [ ] 对话工作台：独立对话界面 + 顶部配置切换栏，可通过对话配置对接系统
- [ ] CLI命令支持：在对话输入框中支持`/`前缀CLI命令，自动补全、历史记录、执行结果展示
- [ ] 配置中心：Agent/Skills/Tools/OAuth卡片列表，CRUD操作正常
- [ ] 文件导入：支持GitHub链接、ZIP上传、MD撰写三种导入方式
- [ ] Agent关联配置：可为Agent选择关联的Skills/Tools/OAuth

### Phase 2 验收
- [ ] 通知中心：按优先级排序的合规事件，点击进入产品对话
- [ ] Toast通知：WebSocket实时推送弹窗
- [ ] 操作执行闭环：确认操作后展示执行计划+进度+结果
- [ ] 产品面板联动：操作完成后自动刷新

### Phase 3 验收
- [ ] 记忆树浏览器支持Obsidian Wiki风格浏览
- [ ] 指标监控中心展示全局和产品级指标趋势
- [ ] Agent状态监控面板实时显示Agent状态
- [ ] Agent配置管理界面支持工具集/Skills配置

### Phase 4 验收
- [ ] 飞书/钉钉消息卡片正常渲染
- [ ] 合规报表支持PDF/Excel导出
- [ ] 集成管理页面展示OAuth连接状态
- [ ] 权限管理支持多角色配置
