## 一、调研结论（决策基础）

### 1.1 现有能力边界

| 能力域 | 现状 | 完成度 |
|--------|------|--------|
| 产品生命周期状态机（8阶段） | ✅ concept→design→sourcing→ready→active→fulfilling→aftersale→end | 100% |
| 产品 CRUD + 合规检查 | ✅ 完整 API | 100% |
| EventBus 事件驱动 | ✅ 事件定义完整，Worker 已注册 | 100% |
| Shopify 商品同步 | ✅ OAuth + 自动拉取 | 90% |
| **选品来源管理** | ❌ 无供应商库、无评分体系、无来源渠道 | 0% |
| **合同生成** | ❌ 无合同模板引擎、无签署流程 | 0% |
| **支付通道** | ⚠️ 仅 Shopify OAuth 骨架，无 Stripe/PayPal/万里汇 | 10% |
| **物流报关** | ⚠️ 仅事件定义，无报关单生成、无物流商 API | 5% |
| 前端生命周期页面 | ⚠️ 仅产品列表卡片（ProductListPage），无流程视图 | 20% |

### 1.2 核心数据缺口

当前 `ProductInfo` 缺少以下字段，需扩展：

```
supply_source      选品来源（1688/工厂直采/Shopify导入）
supplier_id        关联供应商 ID
contract_id        关联合同 ID
payment_channel    支付通道配置（Stripe/PayPal/连连支付）
logistics_info     物流信息（承运商/运单号/报关状态）
customs_status     报关状态（pending/declared/cleared/exception）
```

### 1.3 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 合同模板引擎 | Jinja2（已在项目依赖中） | 无需新增依赖，模板存 `data/contracts/templates/` |
| 物流追踪 | 17TRACK API / AfterShip API（webhook 模式） | 按接入商优先级降级 |
| 支付通道检测 | HTTP 连通性测试 + Webhook 签名验证 | 不存储真实凭证，仅存配置元数据 |
| 存储方式 | JSON 文件（与现有 ProductStorage 保持一致） | 无需新增数据库，支持 OS 级隔离 |

## 三、数据模型

### 3.1 SupplierInfo（供应商）

```python
class SupplierInfo(BaseModel):
    id:              str           # supplier_{uuid8}
    name:            str           # 供应商名称
    source_type:     str           # '1688' | 'factory' | 'platform' | 'overseas'
    contact_name:    str
    contact_phone:   str
    contact_email:   str
    address:         str
    country:         str           # ISO 两字母
    business_license:str           # 营业执照号
    tax_id:          str           # 税号（支持出口退税：has_invoice=True）
    has_invoice:     bool          # 能否开增值税专用发票
    certifications:  list[str]     # ["ISO9001", "CE", "ROHS"]
    categories:      list[str]     # 供应品类 HS 编码前缀
    rating:          float         # 0.0-5.0，人工/AI 综合评分
    risk_level:      str           # 'low' | 'medium' | 'high'
    status:          str           # 'active' | 'suspended' | 'blacklisted'
    tags:            list[str]
    metadata:        dict          # 扩展字段（1688 shop_id 等）
    created_at:      str
    updated_at:      str
```

**存储路径**：`data/suppliers/{supplier_id}/supplier.json`

## 五、合同模板设计

### 5.1 内置模板

| 模板 ID | 名称 | 适用场景 |
|---------|------|---------|
| `purchase_standard` | 标准采购合同 | ToC 零售商品采购 |
| `oem_factory` | 工厂代工合同 | OEM/ODM 生产委托 |
| `logistics_freight` | 货代服务合同 | 货代/物流委托 |
| `nda_supplier` | 保密协议 | 与供应商分享产品设计前 |

### 5.2 合同生成流程

```
用户填写变量（产品名/金额/交货期/贸易术语）
         ↓
Jinja2 模板渲染 → 合同草稿 HTML
         ↓
ZhipuAI LLM 审查：
  - 交货条款合规性（EXW/DDP 风险提示）
  - 价格条款完整性（定金比例/尾款条件）
  - 责任条款完整性（质量保证/违约赔偿）
  - 强制合规条款缺失检测
         ↓
compliance_issues + compliance_score 写回
         ↓
用户确认 → 签署 → 状态更新 → 发布 contract:signed 事件
```

### 5.3 关键合规检查点（LLM Prompt 维度）

```
1. 贸易术语风险：
   EXW → 卖家只负责出厂，买家承担全程物流和清关风险
   DDP → 卖家全包，需确认是否有目的国进口资质
   重点关注：合同中的交货点是否与实际业务承受能力匹配

2. 价格条款：
   - 是否明确汇率约定（固定汇率/浮动汇率/签约日汇率）
   - 定金比例（建议 20-30%，过高/过低都有风险）
   - 尾款触发条件是否清晰（见提单？货到付款？）

3. 质量条款：
   - 是否有验货条款（第三方 SGS/BV 检验）
   - 质保期限、退货条件

4. 禁止出口条款：
   - 是否包含受管制物品（军民两用、稀土材料）
   - 是否涉及制裁国家/地区
```

## 六、物流报关流程

### 6.1 报关单生成逻辑

```
输入：product_id + 目的国 + 申报价值 + 数量
  ↓
Step 1: 从 ProductInfo 获取 hs_code
Step 2: 查询关税税率表（data/customs/tariff_rates.json）
Step 3: 计算应缴关税 = 申报价值 × 关税税率
Step 4: 检查 IOSS 适用性（欧盟目的国 + 订单价值 < €150）
Step 5: 生成申报要素（申报品名映射、原产地默认 CN）
Step 6: LLM 合规审查：
        - HS 编码与品名是否匹配
        - 申报价值是否合理（低申报检测）
        - 是否需要出口许可证（管制品检测）
        - 三单一致性预检
输出：CustomsDeclaration 对象
```

### 6.2 三单一致性检查

```
订单信息（product + qty + value）
    ×
支付信息（payment_channel + amount + currency）
    ×
物流信息（carrier + tracking + declared_value）
          ↓
LLM 对比三者是否一致 → 不一致项列表 → compliance_checks
```

### 6.3 关税税率数据源

| 来源 | 覆盖范围 | 更新频率 |
|------|---------|---------|
| `data/customs/tariff_rates.json` | 本地缓存（EU/US/UK/JP/KR 常用税率） | 手动更新 |
| WTO API（已有 WTOCollector） | 实时查询 | 实时 |
| 米塔搜索（兜底）| 模糊税率信息 | 实时 |

## 九、合规风险清单（嵌入 LLM Prompt）

### 供应商审核维度

1. 营业执照有效期（是否在期）
2. 增值税专用发票资质（出口退税关键）
3. 是否涉及制裁名单（OFAC/EU/UN）
4. 产品认证匹配度（其供应的品类是否有对应认证）
5. 新疆棉/稀土受控材料排查

### 合同核心风险

1. **贸易术语风险**：EXW 卖家责任最小，DDP 风险最大
2. **低申报陷阱**：合同价格与实际申报价值不符
3. **管制品条款缺失**：军民两用物品未声明
4. **汇率风险条款**：未约定汇率锁定机制
5. **知识产权条款**：OEM 产品的知识产权归属未明确

### 报关合规风险

1. **HS 编码错误**：归类错误导致关税税率差异（最高差 20%+）
2. **低申报**：申报价值 < 实际价值（海关可处 2-5 倍罚款）
3. **IOSS 缺失**：欧盟单笔 <€150 必须提供 IOSS，缺失导致买家二次缴税
4. **出口许可证缺失**：管制品未申请许可证
5. **三单不一致**：订单/支付/物流信息不匹配

### 支付通道合规

1. **拒付率**：超过 0.8% 触发 Stripe/PayPal 冻结
2. **KYC 未完成**：账号主体信息未验证
3. **资金链路**：个人账户收款超限（外汇管理局监控）
4. **PCI DSS**：明文存储卡号（禁止）

## 十、接口联动示意

```
用户在 ProductJourneyPage 点击「生成合同」
         ↓
POST /api/v1/contracts/generate
  {product_id, supplier_id, template_id, vars: {delivery_term, amount, ...}}

         ↓
```
